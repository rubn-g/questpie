# RFC: Unified Module Architecture — File-System-as-Source-of-Truth

> Status: **Approved**
> Authors: @drepkovsky
> Date: 2026-02-24
> Supersedes: RFC-FILE-CONVENTION.md §12 (`config()`), §13 (`module()`), §14 (Codegen Plugins)

---

## 1. Summary

Unify the module/config/app architecture so that:

1. **The file system is the sole source of truth** for entity definitions. No hand-written definition objects.
2. **Modules are static** — plain objects of entities. No factory functions, no options, no dynamic behavior. Generated by codegen from the file system.
3. **Plugins are codegen extensions** registered at the config level, not inside modules. They discover new file types and extend the generated output.
4. **Config is runtime + plugins** — `questpie.config.ts` holds only runtime infrastructure (db, adapters, secret) and plugin registrations.
5. **Every level gets codegen** — npm packages run codegen during build, user-land modules during dev, the root app during dev. The generated output is the typed module/app.

---

## 2. Core Concepts

| Concept    | What it is                                                           | User-authored?                  | Example                            |
| ---------- | -------------------------------------------------------------------- | ------------------------------- | ---------------------------------- |
| **Module** | Static plain object of entities (collections, functions, jobs, etc.) | No — GENERATED from file system | `adminModule`, `billingModule`     |
| **Plugin** | Codegen extension — discovers new file types, extends template       | Yes — authored by package/user  | `adminPlugin()`, `openapiPlugin()` |
| **Config** | Runtime (db, adapters, secret) + plugins + CLI settings              | Yes — `questpie.config.ts`      | `runtimeConfig({...})`             |

### 2.1 What Changed from the Old Architecture

| Before                                                    | After                                                                                         |
| --------------------------------------------------------- | --------------------------------------------------------------------------------------------- |
| `module()` factory function with bundled entities         | Static object generated by codegen                                                            |
| `config()` mixes entity config + runtime                  | `runtimeConfig()` is runtime + plugins only                                                   |
| `admin({ branding, sidebar, dashboard })` dynamic factory | Static `adminModule` + user files (`branding.ts`, `sidebar.ts`) discovered by `adminPlugin()` |
| `audit({ dashboard: false })` dynamic factory             | Static `auditModule` — options become plugin-discovered files or config                       |
| Plugins registered inside modules                         | Plugins registered at config level                                                            |
| `createApp(config, entities)` called by codegen           | `createApp(definition, runtime)` — internal, called by generated code only                    |

## 3. User-Facing API

### 3.1 Functions

| Function                         | Purpose                                                   | Who uses it                    |
| -------------------------------- | --------------------------------------------------------- | ------------------------------ |
| `runtimeConfig<T>(input: T): T`  | Identity function for type safety in `questpie.config.ts` | User                           |

## 4. Plugin Resolution Patterns

**This is critical.** Plugins discover files from the file system. The resolution must support ALL of these patterns:

### 4.1 Directory with Default Exports

```
collections/posts.ts     → export default collection("posts")
collections/pages.ts     → export default collection("pages")
```

Resolution: key derived from filename.

```ts
{ posts: import("./collections/posts").default, pages: import("./collections/pages").default }
```

### 4.2 Directory with Named Exports

```
blocks/hero.ts           → export const heroBlock = block("hero")
blocks/cta.ts            → export const ctaBlock = block("cta")
```

Resolution: key derived from export name (not filename).

```ts
{ heroBlock: import("./blocks/hero").heroBlock, ctaBlock: import("./blocks/cta").ctaBlock }
```

### 4.3 Single File with Default Export

```
sidebar.ts               → export default ({ s, c }) => ({...})
```

Resolution: single value, key is the concern name.

```ts
{
	sidebar: import("./sidebar").default;
}
```

### 4.4 Single File with Named Exports

```
blocks.ts                → export const heroBlock = ...; export const ctaBlock = ...
```

Resolution: each named export becomes a separate entry.

```ts
{ heroBlock: import("./blocks").heroBlock, ctaBlock: import("./blocks").ctaBlock }
```

### 4.5 Directory File with Named Exports

```
blocks/hero.ts           → export const heroBlock = ...; export const heroBanner = ...
```

Resolution: all named exports are collected. Note: this means one file can contribute multiple entities.

> **Implementation note:** Codegen currently picks the FIRST named export per file for
> core categories (collections, globals, functions, jobs, blocks). The "resolve: all"
> pattern described here is not yet implemented for those categories.
> Use separate files for separate entities, or the module `functions` object for bundles.

### 4.6 Plugin Discover API

```ts
interface CodegenPlugin {
	name: string;

	/**
	 * Register file patterns to discover.
	 * Key = state key (e.g. "blocks").
	 * Value = pattern definition.
	 */
	discover?: Record<string, DiscoverPattern>;

	/**
	 * Post-discovery hook — modify the generated code.
	 */
	transform?: (ctx: CodegenContext) => void;
}

type DiscoverPattern =
	| string // shorthand: "blocks/*.ts" or "sidebar.ts"
	| {
			pattern: string; // glob pattern
			resolve: "default" | "named" | "all" | "auto";
			/**
			 * "auto" (default): detect from file content
			 *   - if file has default export → use default import
			 *   - if file has named exports → use named imports
			 * "default": always use default import
			 * "named": always use named imports (all exports)
			 * "all": collect all exports (both default and named)
			 */
			keyFrom?: "filename" | "exportName";
			/**
			 * "filename" (default for directory patterns): derive key from file name (camelCase)
			 * "exportName" (default for named exports): derive key from the export identifier
			 */
			cardinality?: "single" | "map";
			/**
			 * "map" (default for directory patterns): creates a Record<string, Entity>
			 * "single" (default for single-file patterns): creates a single value
			 */
	  };
```

When `discover` value is a `string`:

- If it contains `*` or is a directory pattern (`"blocks/*.ts"`): treated as directory pattern with `resolve: "auto"`, `keyFrom: "filename"`, `cardinality: "map"`
- If it's a single file (`"sidebar.ts"`): treated as single-file pattern with `resolve: "auto"`, `keyFrom: "filename"`, `cardinality: "single"`

## 5. Routes — Raw HTTP Handlers

### 5.1 Motivation

The existing `mode: "raw"` functions are limited:

- POST-only (all functions are POST)
- Always under `/rpc/*` path
- No HTTP method routing

For webhooks, payment callbacks, OAuth redirects, and custom API endpoints, we need proper route registration with:

- Custom HTTP methods (GET, POST, PUT, DELETE, PATCH, etc.)
- Custom URL paths
- Direct access to `Request` → `Response`

### 5.2 File Convention

```
routes/
├── webhooks/
│   └── stripe.ts        → POST /api/webhooks/stripe
├── health.ts            → GET /api/health
└── oauth/
    └── callback.ts      → GET /api/oauth/callback
```

### 5.3 Route Definition

```ts
import { route } from "questpie";

// Single method
export default route({
	method: "POST",
	handler: async ({ request, app, session }) => {
		const body = await request.text();
		const sig = request.headers.get("stripe-signature");
		// verify webhook signature...
		return new Response("ok", { status: 200 });
	},
});

// Multiple methods
export default route({
	method: ["GET", "POST"],
	handler: async ({ request, app }) => {
		if (request.method === "GET") {
			return new Response(JSON.stringify({ status: "ok" }));
		}
		// handle POST...
	},
});

// All methods (default if method is omitted)
export default route({
	handler: async ({ request, app }) => {
		return new Response("ok");
	},
});
```

### 5.4 Route Handler Args

```ts
interface RouteHandlerArgs<TApp = any> {
	request: Request;
	app: TApp;
	session: Session | null;
	locale: string;
	db: Database;
	params: Record<string, string>; // URL params if pattern-matched
}
```

### 5.5 URL Mapping

Routes are mapped from file path to URL path:

- `routes/webhooks/stripe.ts` → `/api/routes/webhooks/stripe` (or configurable base path)
- File path segments become URL segments
- The `createFetchHandler` routes to these before the catch-all collection CRUD

### 5.6 Routes in Codegen

Routes are a core file convention:

```ts
// .generated/index.ts
import _route_webhooks_stripe from "../routes/webhooks/stripe";
import _route_health from "../routes/health";

// In the definition:
routes: {
	"webhooks/stripe": _route_webhooks_stripe,
	"health": _route_health,
}
```

## 6. OpenAPI as a Plugin

### 6.1 Current State

OpenAPI is currently a separate package (`packages/openapi`) that provides:

- `withOpenApi(app)` middleware wrapper
- Scalar UI serving
- Spec generation from collections + functions

### 6.2 Vision

OpenAPI should be a **plugin** registered in `questpie.config.ts`:

```ts
import { runtimeConfig } from "questpie";
import { adminPlugin } from "@questpie/admin/plugin";
import { openapiPlugin } from "@questpie/openapi";

export default runtimeConfig({
	plugins: [adminPlugin(), openapiPlugin()],
	app: { url: process.env.APP_URL! },
	db: { url: process.env.DATABASE_URL! },
	// ...
});
```

### 6.3 What the OpenAPI Plugin Enables

1. **Static spec generation at build time** — codegen can produce `openapi.json` alongside the generated code
2. **TypeScript compiler integration** — use the TS compiler to extract function input/output types for spec generation, even without explicit Zod schemas
3. **Route inclusion** — with the new `routes/` convention, raw HTTP routes can also be documented in the spec
4. **Spec serving via a generated route** — the plugin's `transform` hook can inject a route that serves the spec (e.g., `GET /api/openapi.json`)
5. **Scalar/Swagger UI** — the plugin can inject a route for the UI page

### 6.4 Plugin Implementation (Sketch)

```ts
export function openapiPlugin(options?: {
	spec?: { path?: string };
	ui?: { path?: string; enabled?: boolean };
}): CodegenPlugin {
	return {
		name: "questpie-openapi",
		// No discover — OpenAPI doesn't discover files
		transform: (ctx) => {
			// Inject a route that serves the OpenAPI spec
			// Inject a route for Scalar UI
			// Can introspect ctx.collections, ctx.functions, ctx.routes
			// to generate the spec at codegen time
		},
	};
}
```

### 6.5 Scope

This RFC only establishes that OpenAPI should become a plugin. The full implementation details (TS compiler integration, static generation, etc.) are deferred to a separate design document.

## 7. Misc Code Patterns — Utils, Services, Shared Code

### 7.1 The Problem

Not everything fits into entity conventions. Projects need:

- **Utility functions** — `slugify()`, `formatCurrency()`, date helpers
- **Service modules** — email templates, payment processing, external API clients
- **Shared types** — common interfaces used across collections/functions
- **Constants** — enum values, config constants

### 7.2 Convention

Misc code lives outside the convention directories. Codegen **ignores** it — it is not discovered or generated. It is imported directly by entity files that need it.

```
questpie/server/
├── collections/           ← codegen discovers
├── functions/             ← codegen discovers
├── utils/                 ← NOT discovered — manual imports
│   ├── slugify.ts
│   ├── format.ts
│   └── date.ts
├── services/              ← NOT discovered — manual imports
│   ├── stripe.ts
│   ├── email-templates.ts
│   └── geocoding.ts
├── types/                 ← NOT discovered — manual imports
│   └── common.ts
├── constants/             ← NOT discovered — manual imports
│   └── index.ts
├── modules.ts
└── questpie.config.ts
```

### 7.3 Usage in Entity Files

```ts
// collections/posts.ts
import { collection } from "questpie";
import { slugify } from "../utils/slugify";

export default collection("posts")
	.fields(({ f }) => ({
		title: f.text({ label: "Title", required: true }),
		slug: f.slug({ label: "Slug", from: "title" }),
	}))
	.hooks({
		beforeChange: [
			async ({ data }) => {
				if (data.title && !data.slug) {
					data.slug = slugify(data.title);
				}
				return data;
			},
		],
	});
```

### 7.4 Discovery Exclusion Rules

Codegen excludes:

- Files/dirs starting with `_` (e.g., `_helpers.ts`, `_shared/`)
- Directories not matching any convention pattern
- `index.ts` files in convention directories (reserved for barrel exports)
- Any path under `utils/`, `services/`, `types/`, `constants/`, `lib/`, `helpers/` — these are well-known non-entity directories

## 10. Type Safety

### 10.1 How Types Flow

1. Entity files export typed builders/objects — exact types preserved
2. Generated code imports them as `typeof` references — zero inference cost
3. Module type extraction via `ExtractFromModuleArray<typeof _modules, K>` walks the tuple recursively
4. `App` type intersects module types with user entity types
5. `as unknown as App` cast bridges runtime to types (sound — codegen guarantees correctness)

### 10.2 Key Type Utilities

```ts
// Extract property K from a single module (recurses into sub-modules)
type ExtractFromModule<M, K extends string> = (M extends {
	modules: infer Sub extends readonly any[];
}
	? ExtractFromModuleArray<Sub, K>
	: {}) &
	(K extends keyof M ? (M[K] extends Record<string, any> ? M[K] : {}) : {});

// Extract from an array of modules (tuple decomposition)
type ExtractFromModuleArray<
	A extends readonly any[],
	K extends string,
> = A extends readonly [infer First, ...infer Rest]
	? ExtractFromModule<First, K> & ExtractFromModuleArray<Rest, K>
	: {};
```

These work because `typeof _modules` preserves the exact tuple type when `modules.ts` exports a literal array.

### 10.3 Why `ExtractModulesProperty` Changes

Previously: `ExtractModulesProperty<typeof _config, "collections">` — extracted from the config object's `modules` property.

Now: `ExtractFromModuleArray<typeof _modules, "collections">` — extracted directly from the modules array import. Simpler, same result.

## 12. Open Questions / Future Work

1. **OpenAPI plugin implementation** — Full design deferred. This RFC only establishes it should be a plugin. TS compiler integration for type extraction, static spec generation, and route documentation need a separate design.

2. **Feature layout support** — `features/billing/collections/*.ts` should work alongside `collections/*.ts`. The current codegen already scans `features/*/`. Verify this works with the new architecture.

3. **Module-level codegen triggers** — For npm packages, codegen runs during `bun run build`. Need to define the exact build step integration (pre-build script? turbo pipeline?).

4. **Admin extension keys (listViews, editViews, components)** — These are admin-specific registrations in the static module. They don't have file convention discovery. Currently hardcoded in the generated module. Consider whether a plugin could discover these too (e.g., `views/table.ts`, `components/icon.ts`).

5. **Hot module replacement** — When a user adds/removes a file during `questpie dev`, codegen regenerates. Ensure this works with the new module + runtime split.

6. **Email templates** — Could be another plugin-discovered convention: `emails/*.ts` or `emails/*.tsx`.
